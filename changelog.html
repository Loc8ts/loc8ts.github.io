v0.23.0

July 19, 2026

Smarter scanning. Quick-scan now recognises a location QR code and takes you straight to that location - no more detour through the camera app - and barcodes you type or import are now checked on entry, with a gentle warning when an EAN/UPC check digit doesn’t add up. Both build on the existing scanner rather than replacing it. This release carries no schema changes.

Action required

  • Re-download location QR codes if you reach Loc8ts under more than one address. A location QR code only opens on the exact address (scheme + host + port) it was created on. If your app runs under several URLs - say a LAN IP and a public domain, or after moving to a new domain - download each location’s QR code again from the address you actually scan on, so the codes open in the app instead of being rejected.

Features

  • Scan a location QR code to open it. Quick-scan now notices when a scanned QR code is a Loc8ts location code (a /locations/{id} URL on the same address) and takes you straight to that location - you stay in Loc8ts instead of being handed off to the camera or QR app. A code from a different address (another installation, or the same app reached over a different host, scheme or port) is rejected with a clear message that names that address - not a misleading “no item found”. Item barcodes and everything else still go through the usual item lookup, unchanged.
  • Two-tier barcode validation. Barcodes you type in or bring in via CSV import are now checked instead of accepted blindly. A basic sanity check (enforced in frontend and backend, import included) requires 4-50 characters and rejects backslashes and control characters. On top of that, a soft check warns you when an all-numeric EAN/UPC/GTIN code (8, 12, 13 or 14 digits) has a wrong check digit - without ever blocking the save, so valid Code 128, QR and internal number codes still go through. Scanned codes are already verified by the scanner itself, so this closes the gap for manual entry and import.

Improvements

  • Clearer barcode input hint. The barcode field placeholder now reads “Enter or scan a barcode…”, pointing at the scan action right in the field and dropping the misleading “(EAN/UPC)” hint - any code type the scanner supports is accepted now.

Maintenance

  • Dependency bumps. PocketBase v0.39.6 → v0.39.7, plus frontend package updates.

v0.22.0

July 12, 2026
First public release

The first public release. Loc8ts is now published on the GitHub Container Registry at ghcr.io/loc8ts/loc8ts, built as multi-arch images for amd64 and arm64 - self-hosting no longer means loading a hand-delivered tarball, you just pull the image. This release carries no schema changes.

Features

  • Public images on GHCR. Release images are published to ghcr.io/loc8ts/loc8ts and can be pulled directly (docker pull ghcr.io/loc8ts/loc8ts:latest). Each release is tagged three ways: latest, the full version (MAJOR.MINOR.PATCH) and the minor series (MAJOR.MINOR). Publishing is automated - CI builds and pushes on a version tag after verifying it matches the VERSION file.
  • Multi-arch builds (amd64 + arm64). The Docker build cross-compiles the Go backend per target platform, so the same image runs on x86 servers and ARM boards alike.
  • Quick-create item from an unknown barcode. When a scanned barcode matches no item, editors and admins get a “Create item” action on the no-match toast that jumps to the new-item form with the scanned code pre-filled. Viewers still fall through to a filtered search.

Improvements

  • Configurable toast action label. Interactive toasts can override the default “Go to related:” heading (e.g. “Want to add it?” on the barcode no-match toast), and the heading is omitted when blank.

Maintenance

  • Dependency bumps. PocketBase v0.39.5 → v0.39.6, plus frontend package updates.

v0.21.21

July 6, 2026

A realtime overhaul. The movement page now always reflects the “now” - incoming and departing batches, depletions, and flags being raised or resolved update live as other users act, so a movement is never planned against stale amounts. Item and location detail pages get the same treatment, and the app refreshes the signed-in user’s own record live so role and profile changes take effect without a reload. Frontend-only - a straight swap with no schema, Docker or environment changes.

Features

  • Live stock on the movement page. /movement subscribes to stock, items and locations and reflects every change in realtime - incoming batches, departing/depleted batches, and flags being raised or resolved - so amounts, batch lists and pickers are never stale while a movement is being planned. A selected batch that is depleted or removed elsewhere clears itself, and refetches are debounced to coalesce bursts.
  • Realtime item & location detail pages. Stock updates are applied incrementally rather than via a full reload: a batch moved to another location drops off the source view, a deleted batch is removed, and updates patch a single row in place. Threshold alerts refresh alongside every change.
  • Live auth-store refresh. A role or profile change for the signed-in user is reflected across the sidebar, account and any current-user view immediately, with clean handling of account removal and disable.

Fixes

  • Deep links & back-nav filters in production. Global prerendering is now off, so URL query params and back-navigation filters stay intact for deployed builds, including dynamic routes.
  • Mobile bottom-bar highlight on stock count. Fixed a route-matching error that left the stock tab highlighted while on the stock-count route.

v0.21.17

July 5, 2026

A navigation and list-state overhaul. Stock count is promoted to a top-level route and, together with movement, moved into the main navigation, backed by an extended sidebar and a new Quick Action Drawer for fast scan / move / count actions on mobile. List pages gain app-wide URL-driven state - search, page and per-page now live in the URL with back-nav-safe behavior. Frontend-only - no schema, Docker or environment changes.

Features

  • Quick Action Drawer. A new mobile action drawer surfaces the common scan, movement and stock-count actions behind a single control, with an integrated barcode scanner that resolves a scanned code straight to an item. The actions are editor/admin only - viewers see a read-only surface.
  • Stock count promoted to a top-level route. Stock count moved out from under movement to its own /stock-count route and, alongside movement, was promoted into the main navigation.
  • App-wide URL-driven list state. A shared helper mirrors search / page / perPage into the URL with identical behavior across every list page. Defaults are omitted to keep URLs clean, state is back-navigation safe, and multiple paginated lists can coexist on one route.
  • Unified re-click reset. Clicking the already-active sidebar entry for a list page resets that page’s query back to defaults - clearing filters, search and pagination in one place.
  • Items list movement shortcut. The movement button on the items list now redirects straight into the movement flow.

Improvements

  • Interactive toasts. Toasts gained optional action links (label + href), a longer display time for actionable toasts, and pause-on-hover.
  • Batches summary always shown. The batches summary on item and location detail pages is no longer hidden when only a single batch exists.

Maintenance

  • Better seed script for local/dev data seeding.

v0.21.10

July 3, 2026
Breaking change

Multi-user support: a full user-management and role system (admin / editor / viewer), transactional email over SMTP, optional public self-signup, and the account and auth flows that go with them - password reset, email verification, email-change confirmation and forced password changes. Rounded out by a UI polish pass and a switch to the Geist typeface.

Action required

Database reset recommended
  • Collection schema changes (database reset recommended). Several collections gained fields, applied automatically on startup: users gains role (admin / editor / viewer), disabled, must_change_password and pending_approval - app_meta gains smtp_configured, signup_enabled and signup_mode. Existing databases keep working, but a reset gives the cleanest schema. (Pre-1.0.0 databases are treated as disposable, so no migration is provided.)

Features

  • User management & roles. A new admin user-management area lets an admin create, edit, disable and delete users, assign a role (admin / editor / viewer), and force a password change. Roles are enforced backend-first - the UI only disables controls and redirects routes for read-only users.
  • Transactional email over SMTP. A new admin SMTP settings page lets you configure a mail server and fire a test send before saving. A shared email service renders and dispatches verification, password-reset, email-change and account-lifecycle mail.
  • Public self-signup (optional). Self-signup is gated on SMTP being configured and supports two modes - self (email verification alone) and admin_approval (an admin confirms after the user verifies).
  • Account & auth flows. New routes cover forgot / reset password, change password, email-verification confirmation and email-change confirmation, plus a reusable password input with strength feedback.

Improvements

  • Refreshed branding & Geist typeface. Refreshed favicons, a new logo asset and an updated web manifest, and the UI typeface moved to Geist (JetBrains Mono retained for numbers and code).
  • Toast overhaul with refined styling and colors.
  • Reports buttons unified across the activity, item, location and stock views, with aligned download icons and a mobile icon-only variant.
  • Categories display behavior reworked and color picking improved.
  • Backend-down screen shown when the backend is unreachable.

Maintenance

  • Dependency updates, including the Geist font packages.

v0.19.6

June 25, 2026

A time-range filter for the activity log - a default 24-hour window with selectable presets and custom ranges that also flows through into reports - alongside locale-aware global number formatting, a stock-count view that reflects threshold status, and a reworked in-app back/forward navigation. A drop-in upgrade with no schema or Docker changes.

Features

  • Activity log time-range filter. The activity feed defaults to the last 24 hours and offers presets (24H / 7D / 1M), a custom date range and an explicit All time option. The selected range carries through to the activity report as a readable filter chip.
  • Global number formatting. Number display is centralised and locale-aware: large values are abbreviated (≥ 10k → k, ≥ 1M → M) with the exact value revealed in a tooltip only when actually abbreviated.
  • Stock count reflects threshold status. The stock-count flow classifies each row against its thresholds (critical / warning / ok / missing / no-threshold) using the same logic that backs reporting.
  • Reworked back/forward navigation. Back navigation steps through real history when available and falls back to the parent route on deep links or fresh tabs - forward navigation returns you to the view you came from.
  • Stray-file safety for backups. The backup listing ignores stray files dropped into the backups folder, only surfacing recognised auto/manual backups.

Fixes

  • Barcode scanner: lazy-load fallback library. The zxing fallback was loaded on every page request - it now loads lazily and only when native barcode detection is unavailable.
  • Activity flag-resolution wording and styling. Resolution entries label their text “Fix” instead of “Reason”, and an initial note below a resolution is dimmed so the fix reads as primary.
  • Activity diff word-wrapping for long note and field changes.

Maintenance

  • Dependency updates across Svelte, SvelteKit and SQLite.

v0.17.4

June 20, 2026
Breaking change

Stock flagging - individual stock batches can now be reported as damaged, defective, problematic, or annotated with a note, with optional photo/document attachments, a quantity to exclude from totals, and a full resolve workflow. Around it sits a reworked activity log that unifies movements, flag actions and item/location lifecycle events into a single feed, in-app notifications for flagged stock, and a dedicated stock-batch detail page.

Action required

Database reset recommended
  • Collection schema changes (database reset recommended). A new activity collection backs the reworked activity log - stock_levels gains flagging fields (status, note, excluded_from_stock, files, flagged_by, flagged_at) - and the users name max length was reduced from 100 to 50. These apply automatically on startup and existing databases keep working, but a reset gives the cleanest schema.
  • Docker: compose file changed - update your compose file before upgrading. APP_NAME was removed (the admin UI name is now fixed to Loc8ts), and a new TZ variable (e.g. Europe/Berlin) sets the local timezone used by scheduled backups (defaults to UTC). Stale compose files still passing APP_NAME will run backups in the wrong timezone.

Features

  • Stock flagging (“Report a problem”). Any stock batch can be flagged from the stock-count flow with a status of Damaged, Defective, Problem or Note, an optional note, a quantity, and image/document attachments. Flags can mark a quantity as excluded from stock and threshold totals, and carry through to the item and location batch lists with colour-coded status pills.
  • Resolve workflow. Flagged batches can be resolved with a reason, closing out the problem and recording a resolve entry in the activity log.
  • Stock-batch detail page. A new /stock/[id] route shows a single batch in full - its status, note, attachments and history.
  • In-app notifications for flagged stock. The home dashboard counts flagged stock (problem statuses only) into its warning totals, so problems are visible without digging into individual batches.
  • Reworked activity log. Movements, flag actions and item/location soft-delete / restore events are unified into a single feed with per-type icons and badges and a richer activity-detail view.
  • Batch sorting for items & locations - flagged batches grouped by severity, then by expiry date.
  • CSV export/import reflects flag status - the CSV workflow round-trips the new flag/status fields.
  • App metadata & schema versioning. Added an app_meta collection and tidied version paths - the first groundwork for future database migrations (post-1.0.0).

Improvements

  • Unified delete behaviour added to categories and thresholds.
  • Filter wording: the Stock filter label “Expiry” was corrected to “Type” in reports.
  • Smooth scrolling added to the items list.

Fixes

  • Timezone fix - scheduled-backup times now respect the configured TZ.

Maintenance

  • Dependency updates and a developer report-test script.

v0.14.2

June 12, 2026
Breaking change

CSV import/export for all core data, scheduled and manual backups, and an application-wide data wipe - plus SKU uniqueness handling, a reusable Toggle component and a category-schema simplification.

Action required

Database reset optional
  • Category schema change (optional database reset). The categories collection no longer defines the self-referencing parent_id field. The change applies automatically on startup and an existing database keeps working, but you may reset for a perfectly clean schema.
  • Docker: new backups mount path. The compose file now optionally mounts a backups volume. If you run via Docker and maintain your own compose file, add the matching path so generated backups persist outside the container.

Features

  • CSV Export / Import. The Admin page gains a full CSV export and import workflow covering items, categories, locations, stock and thresholds. Downloadable templates are provided for each, the import shows a diff of what will change before you commit, and a no-change import is blocked.
  • Backups. Create application backups on demand and schedule automatic ones from both the Setup wizard and the Admin page. Schedules support daily / weekly / monthly / every-N-months / custom-cron frequencies with a configurable hour, retention limit and manual vs. auto types.
  • Application-wide wipe. Admins can wipe all application data in one action. The wipe clears every collection except user accounts and removes stored files.
  • SKU uniqueness handling. SKUs are slugified and guaranteed unique - auto-generation falls back cleanly when a value collides.
  • Reusable Toggle component replaces hand-rolled toggle markup across the category, item, location, movement, stock-count and thresholds screens.

Improvements

  • Categories simplified - removed the unused parent_id relation from the collection, API types and UI.

Fixes

  • Backups filesystem handle is now closed with proper error handling.

Maintenance

  • Dependency updates - PocketBase 0.39.1 → 0.39.3.

v0.10.1

June 4, 2026
Breaking change

A pagination overhaul, a security hardening pass on user roles and account passwords, a new “count another” flow for stock counts, and search robustness around the backslash character.

Action required

Database reset optional
  • Database reset (optional). This release tightens the users collection update rules and adds write-validation that forbids the backslash character (\) in item names and barcodes. The rule change applies automatically on startup - any item or barcode created earlier that contains a \ becomes non-compliant. A reset is only needed if existing items contain a backslash.

Security

  • Privilege-escalation fix - users can no longer self-assign a role. The users update rule now rejects any request where a non-admin tries to change their own role or verified field - those may only be changed by an admin.
  • Same-password guard on account. Changing your account password now rejects re-submitting the current password, so a “change” can’t silently be a no-op.

Features

  • Pagination overhaul. The shared pagination component was reworked and rolled out across the Home, Items, Item detail, Activity, Stock, Thresholds and Location detail screens.
  • Stock Count - “Count another”. A new toggle lets you save a stock count and immediately start a fresh one without leaving the screen.
  • Clearer search feedback for invalid input. Typing a backslash into an item / SKU / barcode search now shows an inline hint instead of silently returning the full list.
  • Backslash validation in item forms. Creating or editing an item rejects a backslash in the name or a barcode, with the error shown under the relevant field.

Improvements

  • Parameterised search filters. Item and stock search bind the search term and category filter through query placeholders instead of interpolating raw input - removing a class of filter breakage.
  • Report sorting and wording updated - the expiry status label “fresh” is now “not expired”.

Fixes

  • Pagination scroll bug on the Activity, Items, Stock and Thresholds pages.

Maintenance

  • Dependency updates across backend and frontend.

v0.9.0

A dedicated stock count workflow for verifying physical inventory against recorded quantities, plus departure-stock visibility on the item detail page and a PDF reporting refactor.

v0.5.9

May 5, 2026
Initial development

The foundation. This entry gathers roughly two months of initial development - from the very first commit to the build first handed to a small group of internal alpha testers. It stands up the whole application: a SvelteKit frontend on a PocketBase (Go) backend with server-side hooks, covering items, hierarchical locations, categories, stock movements and thresholds, a dashboard with a live activity feed, barcode and QR-code support, realtime updates, and Docker packaging.

Features

  • Items. Create, edit and delete items with categories, images and file attachments, automatic SKU generation, multiple barcodes per item, current stock at a glance, and a restore flow for deleted items.
  • Hierarchical locations. Nest locations as deep as needed and browse them through a collapsible tree picker, attach images and files, generate per-location QR codes (with a dark-mode variant), and page and search through the items held at each.
  • Categories. Create and edit categories with validation, plus an automatic “Uncategorized” fallback.
  • Stock movements. Book stock in, out and between locations, enforced by backend hooks, with expiry handling and before/after quantity snapshots on every movement.
  • Stock thresholds. Per item/location critical and warning levels with restock actions, current-stock display and stale-count warnings.
  • Dashboard & activity feed. A dashboard backed by server-side aggregation, a full activity feed with per-entry diffs, and an app-wide warning and notification system.
  • Barcodes & QR codes. App-wide barcode validation and duplicate detection, multiple barcodes per item, a camera barcode scanner, and QR codes to jump straight to a location.
  • Realtime updates. Live subscriptions keep items, locations and movements current across sessions, with edit guards so in-progress changes are never clobbered.
  • Accounts & first-run setup. A guided setup with superuser/admin creation, login and basic user handling, and an account and admin area.

Improvements

  • App-wide UI building blocks. Shared toast notifications, pagination, search-input and lightbox components, a loading spinner, dark mode, and mobile navigation with a swipe-to-open sidebar.
  • Editing safeguards. Saving is blocked when nothing changed, edits are cancelled on navigation, and save/cancel behaves consistently across forms.
  • Dates & formatting. A reworked date picker and timezone-aware date handling.
  • Self-hosted assets & polish. Local fonts instead of CDN delivery, plus an ongoing visual and layout polish pass.

Fixes

  • Early-development stability pass. A steady stream of fixes across edit guards, deletion persistence, unique-name indexing, theme flicker, and layout and word-wrapping.

Maintenance

  • Project scaffolding. The initial SvelteKit frontend and PocketBase backend, the database model, README/LICENSE, a Makefile, and Docker image builds via the static adapter.
  • Dependency setup and refactors across backend and frontend as the codebase took shape.
